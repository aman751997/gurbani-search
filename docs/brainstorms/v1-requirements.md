# Gurbani Semantic Search — v1 Requirements

**Status:** Requirements drafted
**Date:** 2026-04-21
**Target ship:** ~10 weeks of evenings/weekends
**Author:** Amandeep Singh

---

## 1. Product one-liner

A semantic search app for the Guru Granth Sahib. Users ask natural-language questions in English or Roman-script Punjabi, and get back real shabads — verbatim, with translations, transliteration, and a short explanation of why each result matches.

**Tagline / brand promise:** *"Finds your Gurbani. Never writes it."* The app retrieves authentic shabads only; it never generates, paraphrases, or summarizes scripture. This is the product's moral core and its primary differentiator — not a footnote, but the homepage.

## 2. Problem and motivation

Every existing Gurbani app (SikhiToTheMax, iGurbani, SearchGurbani) offers keyword-only search. Typing "forgiveness" surfaces shabads containing the literal word "forgiveness" in an English translation. There is no way to ask "what does Gurbani say about ego?" and get the shabads that actually address *haumai*, regardless of which English words translators used.

The one prior AI attempt (KhalsaGPT) used generative modeling and was caught fabricating Gurbani — which is why the SGPC has an active AI sub-committee and why the community drew a clear line: **retrieval of authentic verses is acceptable; generation of scripture is not.**

The product opportunity is the gap between these two: concept-level retrieval over canonical, community-vetted Gurbani, with zero generation of scripture.

## 3. North-star use case

> "I'm wondering what Gurbani says about anger. I open the app, type 'anger', and get back 10 shabads that actually address krodh — ranked by meaning, shown in Gurmukhi with transliteration and English, with a one-line explanation of why each one matches. I can click through to read the full shabad on its Ang."

If v1 nails this flow and nothing else, v1 is a success.

## 4. In scope

V1 ships as two explicit milestones. **v1.0 is the honest v1** — if only v1.0 ships, the project is a success per Section 3. **v1.1 is polish** — it ships only if v1.0 ships on time and there is real time left.

**v1.0 — north-star only** (Sections 4.1, 4.2, 4.3 minus adjacent shabads, 4.6)
**v1.1 — polish tier** (Sections 4.3 adjacent shabads, 4.4, 4.5, Section 7.3 Langfuse dashboard)

### 4.1 Core search experience *(v1.0)*

- **Semantic search** over the Sri Guru Granth Sahib (~6,000 shabads) using BGE-M3 embeddings with multi-view indexing (one vector per shabad for Gurmukhi, one for English translation; Hindi deferred to v2).
- **Query languages accepted:** English and Roman-script Punjabi (e.g. `sat naam`, `haumai`, `simran`). Script detection routes Roman-Punjabi through Aksharamukha transliteration before embedding.
- **Homepage:** thematic starter-query grid (`anger`, `seva`, `ego`, `death`, `devotion`, `forgiveness`, `fear`, `love`, `doubt`, `truth`) so any visitor — including a recruiter — can click once and see the product work.
- **Result ranking:** top 10 shabads by cosine similarity, merged across Gurmukhi + English vector views with dedupe-by-max-score.

### 4.2 Result card (per shabad) *(v1.0)*

Each result shows:

- Gurmukhi text (verbatim, never paraphrased)
- Roman-script transliteration
- English translation (Sant Singh Khalsa)
- Ang number, author (Mahalla), raag
- **"Why this matches"** — a 1-2 sentence caption generated by Claude 4.5 Haiku that explains the semantic connection to the query. Bound by the permanent constraint in Section 6: must never paraphrase, summarize, or quote beyond a single phrase from the shabad. A "single phrase" here means a single contiguous sequence of up to 5 tokens from the Gurmukhi, transliteration, or translation — no stitching multiple phrases into the caption.
  - **Architectural hardening (v1.0):** caption is rendered in a visually and structurally distinct component from the scripture fields — different component boundary, different background, explicit `AI explanation — not scripture` label. Caption text is stored and queried from a different database column from shabad text; UI templates never accept a free-form string in the scripture slot.
  - **Pre-computed captions for starter queries:** captions for each of the 10 homepage starter queries × top-10 results (100 total) are pre-generated offline and committed to the repo as static JSON. These are the only captions manually reviewed for v1.0 launch.
  - **Live-query captions** are generated at request time and cached server-side keyed by (query, shabad_id). First-seen (query, shabad) pairs have a ~1–2s latency budget for caption generation; cache hits are instant. Cached captions are spot-checked in a weekly sample of ~50 for non-paraphrasing compliance for the first month after launch.

### 4.3 Shabad detail page (clickthrough) *(v1.0 minus adjacent-shabads)*

- **v1.0:** Full shabad with all the above fields. Per-shabad shareable URL (`/shabad/{id}`).
- **v1.1:** Adjacent shabads on the same Ang shown as context.

### 4.4 Filters *(v1.1)*

- Filter results by author (Guru Nanak, Guru Arjan, etc.) and raag.
- Implemented as SQL predicates against pgvector query.

### 4.5 Offline PWA *(v1.1)*

- Installable to home screen on iOS and Android.
- Service worker caches the shabad corpus (~10MB) so previously-viewed shabads remain readable offline.
- Recent 20 searches cached. New semantic searches require network (embedding generation is server-side).
- Only cached captions (from server-side cache bundled into the PWA payload) are shown offline; no caption is generated on-device.

### 4.6 Baseline quality bar *(v1.0 — assumed, not a "feature")*

- Bundled Noto Sans Gurmukhi font for deterministic script rendering across iOS, Android, desktop.
- Light + dark mode via `prefers-color-scheme`.
- Font-size control for Gurmukhi readability.
- Every result and shabad has a shareable URL.
- Page loads under 1s on 4G; search round-trip under 2s p95.

## 5. Non-goals for v1 (explicitly deferred)

These are deliberate cuts, not oversights:

- **User accounts, bookmarks, journal, notes.** v1 is fully anonymous. Shareable URLs substitute for "save for later."
- **Hindi / Gurmukhi-script query input.** English + Roman Punjabi only in v1.
- **Audio playback** of recitations. Integration complexity too high for 10-week budget.
- **Shabad-of-the-day / Hukamnama / daily companion flows.** Different product surface.
- **"Related shabads" strip** on result cards. Shifted to v2; implementable cheaply later via a second vector query.
- **Guided learning themes / curated collections.** Adjacent product; out of scope.
- **Commentary / arth / teeka** from historical interpreters. Potentially valuable for v2 but carries its own licensing and curation workload.
- **Fine-tuning the embedding model** on Gurbani text. Interesting research project but not needed for v1 quality.

## 6. Non-goals forever (hard limits)

These will never be features of this product regardless of version:

- **No generation, paraphrasing, or summarization of Gurbani.** The LLM's only scripture-adjacent job is explaining why a retrieved shabad matches a query, and even that is constrained to non-paraphrasing language.
- **No authoritative interpretation (*arth*).** The UI will explicitly frame results as retrieval, and direct users to a granthi or published commentary for meaning.
- **No claims of spiritual authority or completeness.** The footer will state the tool is assistive, not authoritative.

## 7. Success criteria

### 7.1 Retrieval quality (transparent target, NOT a hard gate)

A hand-curated 75-query gold set covering English and Roman-Punjabi inputs, with 3-5 known-relevant shabads per query. Targets:

- **nDCG@10 ≥ 0.70** (target)
- **MRR@10 ≥ 0.55** (target)
- **Recall@20 ≥ 0.85** (target)

These are reported honestly in the README regardless of what the numbers come out to be. An honest 0.62 with a clear writeup beats an aspirational 0.70 that gates shipping. Numbers may be re-baselined after initial eval.

### 7.2 Qualitative quality (HARD gate for v1.0)

Ten hand-picked query/result pairs on the homepage as a "see it work" showcase. Every caption on every starter-query result is hand-reviewed before v1.0 launch for non-paraphrasing compliance. This is the single blocking gate for v1.0 — if any caption paraphrases scripture, v1.0 does not ship until fixed. Live-query captions generated after launch are spot-checked weekly (sample ~50) for the first month.

### 7.3 Portfolio quality (soft goal)

**v1.0:**
- Real custom domain, production-grade TLS.
- README with architecture diagram, eval methodology, and honest eval numbers.
- Homepage so that a recruiter can load it, click a starter query, see a ranked shabad with a caption, and grok the product in 30 seconds.

**v1.1:**
- Langfuse (or equivalent) traces public on a demo dashboard.

### 7.4 Community quality (soft goal)

- Every result links to its Ang so users can verify independently.
- Footer acknowledges BaniDB attribution and Sant Singh Khalsa translation.
- No community-facing claims of completeness or authority.

### 7.5 Learning goals (soft goal, for the creator)

Explicit capabilities this project must build in the creator, used as a tiebreaker when scope pressure forces cuts:

- **Evaluation-driven retrieval system design** — building and interpreting a proper eval harness (nDCG / MRR / Recall) over a hand-curated gold set is a transferable skill; preserve the eval work even if retrieval numbers are modest.
- **Cross-lingual embedding model selection** — making and defending a reasoned choice between BGE-M3, Cohere, and hosted alternatives with published benchmarks.
- **LLM-in-the-loop product design with hard safety constraints** — demonstrating when *not* to use generation and how to architect defense-in-depth around a bright line (never produce scripture).

## 8. Open questions for planning

These are intentionally deferred to `/ce:plan` and do not belong in this document:

**Data and licensing:**
- BaniDB Alliance partner application process and timing. If approval is uncertain at W1, what fallback corpus source is used (e.g., BaniDB open JSON dumps, GurbaniNow, SikhiToTheMax desktop data files)?
- Sant Singh Khalsa translation — confirm redistribution terms before bundling into the offline PWA cache. Is a second translation (public-domain Bhai Manmohan Singh) held as fallback?

**Infrastructure (near-zero-cost stack, ~$1/month total):**
- Embeddings: **Cloudflare Workers AI** (`@cf/baai/bge-m3`) on the free tier (10,000 neurons/day).
- Vector DB: **Supabase free tier** (Postgres + pgvector, 500MB).
- App + API: **Next.js API routes on Vercel hobby tier**. No separate FastAPI service; no Fly.io. Python only used locally on the creator's laptop for one-time corpus ingestion and embedding generation.
- LLM captions: **Claude 4.5 Haiku via Anthropic API** — pre-computed captions for the 10 starter queries × top-10 results are committed as static JSON; live-query captions generated at request time and cached keyed by (query, shabad_id). Expected cost <$1/month at portfolio traffic levels.
- Domain: `.app` or `.dev`, ~$10/year.
- Open: exact pgvector index configuration (HNSW vs IVFFlat) and hybrid search weight. Resolved in planning.
- Open: exact prompt structure and safety rails for the caption generator, including the render-path separation contract between caption field and scripture fields.

**Evaluation:**
- Whether the 75-query gold set is authored solo or pulled from a Sikh-community beta group. Evaluator bias is a known risk.
- Whether the retrieval quality numbers in Section 7.1 are treated as a hard gate, or as transparently-reported targets (with a soft gate only on qualitative showcase queries).

**Unresolved UX states (to be designed in implementation):**
- Loading states for search and caption generation.
- Error states for network / embedding / caption failure.
- Empty states for zero results and zero-after-filters.
- Offline search-attempt state.
- First-run orientation that communicates this is semantic (not keyword) search.
- Responsive breakpoints for the starter-query grid, filter UI, and adjacent-shabads layout.
- Gurmukhi typography details — minimum font size on mobile, line-height for the *shirorekha*, screen-reader `lang` attributes, visual separation between caption field and translation field.

## 9. Timeline (rough)

Two milestones. v1.0 is the honest release; v1.1 is polish that ships if there's real time left.

**v1.0 — target W6–7:**
- **W1:** Local corpus ingestion (BaniDB or fallback source). BGE-M3 embeddings generated locally on laptop, loaded into Supabase pgvector. No cloud inference infra yet.
- **W2:** Next.js app scaffold. Cloudflare Workers AI integration for query-time embeddings. 75-query gold-set authored + eval harness implemented.
- **W3:** Retrieval endpoint + multi-view merge (or single-view baseline, whichever eval supports) + eval run. Reranking iteration if needed.
- **W4:** Result cards with Gurmukhi font pipeline + transliteration + pre-computed starter-query captions + architectural separation of caption/scripture fields.
- **W5:** Homepage with thematic starter-query grid + tagline. Shabad detail page (without adjacent shabads). Light/dark mode + font-size control.
- **W6:** Manual hand-review of every starter-query caption. Custom domain, TLS, README with architecture diagram + eval numbers. **v1.0 ships.**

**v1.1 — target W8–10 if W6 ships on time:**
- **W7:** Adjacent-shabads display on detail page + filters (author + raag).
- **W8:** Offline PWA (service worker, corpus cache, install prompt).
- **W9:** Langfuse traces + public observability dashboard.
- **W10:** Final polish, a short video demo, README updates. **v1.1 ships.**

If any week slips, v1.1 features are cut in order from bottom to top (Langfuse → offline PWA → filters → adjacent shabads), not borrowed against v1.0 time.

## 10. Definition of done

### v1.0 (the honest v1 — project succeeds if this ships)

- [ ] Search works for all 10 homepage starter queries.
- [ ] Every caption on every starter-query result manually reviewed for non-paraphrasing.
- [ ] Caption field is structurally separated from scripture fields in the rendering pipeline (different component, different DB column, different template slot).
- [ ] Homepage communicates "retrieval only, never generation" as the tagline, not a footer disclaimer.
- [ ] Retrieval metrics (nDCG/MRR/Recall) honestly reported in README regardless of value.
- [ ] Deployed to a custom domain with production TLS.
- [ ] README includes architecture diagram + eval numbers + three example queries with screenshots.

### v1.1 (polish — only if v1.0 shipped on time)

- [ ] Adjacent-shabads display on the detail page.
- [ ] Author + raag filters.
- [ ] PWA installs on iOS + Android; corpus reads offline.
- [ ] Public Langfuse dashboard.
- [ ] Short video demo in README.
